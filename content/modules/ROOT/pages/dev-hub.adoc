== Module Overview

**Duration:** 15-20 minutes +
**Format:** Demo + Discussion +
**Audience:** IT Operations, Infrastructure Architects

== Learning Objectives

By the end of this module, participants will understand:

* What Developer Hub (Red Hat's Backstage) is and why Ops should care
* How self-service portals reduce operational toil
* The relationship between Ops governance and developer autonomy
* How to support developers while maintaining control

== Content Outline

=== Part 1: The Developer Experience Problem (5 mins)

==== Traditional Model (Pre-DevHub)
**Developer asks Ops:**
* "Can I have a database?"
* "Can you create me a namespace?"
* "What's my Jenkins URL?"
* "How do I deploy to production?"

**Ops bottleneck:**
* Ticket queue grows
* Developers wait days
* Ops team overloaded with repetitive tasks

==== The Self-Service Solution
* **Developer Hub:** Internal Developer Portal
* Catalog of services, templates, and documentation
* Developers provision what they need (within guardrails)
* Ops defines what's available

=== Part 2: What is Red Hat Developer Hub? (5 mins)

==== Based on Backstage (CNCF)
* Open source developer portal platform
* Created by Spotify, now CNCF project
* Red Hat productized version = Developer Hub

==== Key Features

**Software Catalog:**
* All applications, APIs, services in one place
* Who owns what
* Service dependencies

**Software Templates (Golden Paths):**
* Pre-approved application scaffolding
* "Create a Spring Boot app" in 3 clicks
* Ops controls the template (security, standards)

**TechDocs:**
* Documentation as code
* Markdown docs alongside app code
* Always up-to-date

**Plugins Ecosystem:**
* Kubernetes plugin - see deployments, pods
* ArgoCD plugin - deployment status
* Tekton plugin - pipeline runs
* And dozens more

=== Part 3: The Ops Perspective - Why You Should Care (5 mins)

==== It's About Governance, Not Access

**Without Dev Hub:**
* Developers ask for "root access"
* Ops says "no" (security concern)
* Deadlock and shadow IT

**With Dev Hub:**
* Ops creates "Golden Path" templates
* Developers self-service *approved* patterns
* Ops maintains control via the template

==== Example: Database Provisioning

**Traditional:**
1. Developer opens ticket: "Need PostgreSQL"
2. Ops manually provisions
3. 3-5 day turnaround
4. Developer doesn't know connection string location

**Dev Hub Approach:**
1. Developer clicks "Provision Database" template
2. Ops-approved template runs:
   * Creates namespace (within quota limits Ops set)
   * Deploys Postgres operator
   * Creates database
   * Stores credentials in Vault
   * Updates service catalog
3. Done in 5 minutes
4. Ops never touched it, but controls the *how*

=== Part 5: Implementation Discussion (5 mins)

==== What Ops Needs to Do

**1. Install Developer Hub**
[source,bash,role="execute"]
----
# Install Red Hat Developer Hub operator
oc apply -f developer-hub-operator-subscription.yaml
----

**2. Create Templates (Golden Paths)**
* Define organization-approved patterns
* Example templates:
  - Microservice (Spring Boot, Quarkus, Node.js)
  - Database
  - Message Queue
  - Full app stack

**3. Integrate with Existing Systems**
* Connect to Git (GitHub, GitLab, Gitea)
* Connect to CI/CD (Tekton, Jenkins)
* Connect to Monitoring (Prometheus)
* Connect to identity provider (LDAP, OAuth)

**4. Governance Configuration**
* What can developers do?
* What needs approval?
* Resource limits per template
